Objects

Everything in Python is an object

What does this mean? What is an object?

In CS, an object is

  • A “package” of values (‘state’) and behaviors (‘methods’)
  • An instance of a class

Value Storage

objects can store values (and different objects have different values).

gus = Dog(name='Gus', owner='Michael')

gus
Dog(name='Gus', owner='Michael')

Value Storage

Access object values using the dot operator

gus.owner
'Michael'

The dot looks inside the object for an attribute called owner.

Methods

In addition to storing values, objects can contain functions.

obj.func(arg1, arg2)

is transformed to

func(obj, arg1, arg2)

For example

gus.bark()
'Woof!'

and

gus.bark(3)
'Woof!Woof!Woof!'

Methods

(Basically) Everything is a method under the hood:

# 2 + 2
x = 2
x.__add__(2)
4

Magic double-underscore (“dunder”) methods should not be used directly, but are implicitly called by most operations

Classes

A class is a “category” that defines a set of behaviors and provides space for values to be stored

object template

Because everything in python is an object, the behavior of objects is controlled by their class

gus.bark()
'Woof!'

gus (object) has no special bark, so fallback to Dog class which defines a bark method

Defining Classes

You can define a class with a class statement:

class Dog:
    def __init__(self, name, owner):
        self.name = name
        self.owner = owner
        
    def __repr__(self):
        return f"Dog(name='{self.name}', owner='{self.owner}')"
    
    def bark(self, n=1):
        return "Woof!" * n

Two ‘magic’ methods

  • __init__: What happens when we initialize a new instance?
    • Value attributes set here
  • __repr__: Default print method of object

Subclasses

Subclasses are hierarchical refinements of classes

X is a subclass of Y if all instances of X are also Y but not vice versa

For example: dog breeds

  • Basset Hound: dog that howls instead of barking
  • Basenji: dog that doesn’t bark
  • Collie: dog with a regular bark

Subclasses

class BassetHound(Dog):
    breed = "Basset Hound"
    
    def bark(self, n=1): 
        return "Arooo!" * n
    
class Basenji(Dog):
    breed = "Basenji"
    
    def bark(self, n=1):
        return ""
    
class Collie(Dog):
    breed = "Collie"

Subclasses

nora = BassetHound(name="Nora", owner="Michael")
nora # Default Dog __repr__ used to print
Dog(name='Nora', owner='Michael')

Subclass uses specialized methods first:

nora.bark()
'Arooo!'

If no method found, fall back to default

rex = Collie(name="Rex", owner="Bernard")
rex.bark()
'Woof!'

Methods vs Functions

Why use methods instead of functions?

  • Avoid name collisions:
    • We don’t need basset_bark(), basenji_bark(), etc
  • Keep definitions ‘inside’ classes
  • Make mutating intent clear

Methods vs Functions

Unlike R, python objects can be changed “in place”:

class BankAccount:
    def __init__(self, initial_balance=0):
        self._balance = initial_balance
    
    def deposit(self, deposit_amount):
        self._balance += deposit_amount
        
    def withdraw(self, withdraw_amount):
        self._balance -= withdraw_amount
        
    def balance(self):
        return self._balance
        
    def __repr__(self):
        return f"BankAccount({self._balance})"

Methods vs Functions

my_account = BankAccount(100)
my_account
BankAccount(100)

Make a new deposit:

my_account.deposit(50)
my_account
BankAccount(150)

Make a withdrawal:

my_account.withdraw(150)
my_account.balance()
0

Mutating Methods

Methods which change an object ‘in place’ usually don’t return any value

x = [4, 1, 2]
x.sort()

No value returned, but afterwards:

x
[1, 2, 4]

Danger - Multiple References

Because python objects can be changed in place, we need to understand references

Two names can refer to same object:

  • “Prof. Weylandt” (to you) => Me
  • “Michael” (to my dept. chair) => Me
  • “Honey” (to my wife) => Me

If “Michael” catches a cold, so does “Prof. Weylandt” and “Honey”

Danger - Multiple References

x = [3, 1, 2]
y = x
print(y)

## Lots of code here...
x.sort()

## More code ...

## Finally
print(y)
[3, 1, 2]
[1, 2, 3]

y got changed because same object as x, even though we never touched y:

x is y
True

Danger - Multiple References

The copy method makes an independent copy:

x is y
True
y = x.copy()
print(x is y)
False

Now changes to x don’t impact y:

x.append(4)
x
[1, 2, 3, 4]
y # No 4!
[1, 2, 3]
